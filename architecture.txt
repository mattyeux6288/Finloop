  1. PRINCIPES DIRECTEURS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ▸ Tout ce qui est lisible par un humain (noms de fichiers, dossiers, variables
    métier) est en FRANÇAIS. Le code technique (React, Tailwind, Knex, etc.)
    conserve ses termes anglais car ce sont des noms propres.

  ▸ Un dossier = une responsabilité unique. On ne mélange jamais les couches.

  ▸ Les fichiers sont nommés selon ce qu'ils FONT, pas selon leur type technique.
    Exemple : "analyses.service.ts" dit clairement "je fournis des analyses",
    pas juste "service.ts" ou "analysisService.ts".

  ▸ Ordre de lecture naturel : de gauche à droite dans l'arborescence, on va
    du plus général (configuration, types) au plus spécifique (vues, services).

  ▸ Règle d'or : un développeur junior doit comprendre où se trouve n'importe
    quel fichier en moins de 10 secondes.

  2. VUE D'ENSEMBLE DU MONOREPO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  finloop/
  ├── packages/
  │   ├── noyau/          ← Types, constantes, utilitaires (partagé par tous)
  │   ├── moteur/         ← Calculs financiers (parseurs, analyseurs, validateurs)
  │   ├── api/            ← Serveur HTTP (routes, services, base de données)
  │   └── interface/      ← Frontend React (vues, composants, état)
  │
  ├── api/                ← Point d'entrée Vercel serverless
  │   ├── [[...route]].ts ← Entrée unique, décore l'app Express
  │   └── sante.ts        ← Endpoint de vérification (/api/sante)
  │
  ├── architecture.txt    ← CE FICHIER
  ├── package.json        ← Configuration workspace npm
  ├── tsconfig.base.json  ← TypeScript commun à tous les packages
  └── vercel.json         ← Déploiement Vercel

  Ordre de build obligatoire : noyau → moteur → api → interface
  (chaque package dépend du précédent)

  ┌─────────────────┬────────────────┬───────────────────────────────────────┐
  │ id              │ UUID (PK)      │                                       │
  │ entreprise_id   │ UUID (FK)      │ → entreprises.id                      │
  │ libelle         │ VARCHAR(100)   │ Ex: "Exercice 2025"                   │
  │ date_debut      │ DATE           │                                       │
  │ date_fin        │ DATE           │                                       │
  │ est_clos        │ BOOLEAN        │ Exercice archivé                      │
  │ cree_le         │ TIMESTAMP      │                                       │
  └─────────────────┴────────────────┴───────────────────────────────────────┘

  TABLE imports
  ┌─────────────────┬────────────────┬───────────────────────────────────────┐
  │ id              │ UUID (PK)      │                                       │
  │ exercice_id     │ UUID (FK)      │ → exercices_fiscaux.id                │
  │ nom_fichier     │ VARCHAR(255)   │ Nom du FEC importé                    │
  │ statut          │ VARCHAR(20)    │ 'succes', 'erreur', 'en_cours'        │
  │ nb_ecritures    │ INTEGER        │ Nombre de lignes importées            │
  │ erreurs         │ JSONB          │ Tableau des erreurs de validation     │
  │ avertissements  │ JSONB          │ Tableau des avertissements            │
  │ cree_le         │ TIMESTAMP      │                                       │
  └─────────────────┴────────────────┴───────────────────────────────────────┘

  TABLE ecritures   (cœur de la comptabilité — peut contenir des millions de lignes)
  ┌─────────────────┬────────────────┬───────────────────────────────────────┐
  │ id              │ UUID (PK)      │                                       │
  │ import_id       │ UUID (FK)      │ → imports.id                          │
  │ exercice_id     │ UUID (FK)      │ → exercices_fiscaux.id (index)        │
  │ code_journal    │ VARCHAR(10)    │ Ex: "VTE", "ACH", "BQ"               │
  │ date_ecriture   │ DATE           │                                       │
  │ numero_compte   │ VARCHAR(20)    │ Numéro de compte PCG (ex: 411000)    │
  │ libelle_compte  │ VARCHAR(255)   │                                       │
  │ piece_ref       │ VARCHAR(50)    │ Numéro de pièce justificative         │
  │ libelle         │ VARCHAR(500)   │ Description de l'opération            │
  │ debit           │ DECIMAL(15,2)  │ Montant au débit                      │
  │ credit          │ DECIMAL(15,2)  │ Montant au crédit                     │
  └─────────────────┴────────────────┴───────────────────────────────────────┘

  TABLE cache_analyses  (évite de recalculer les états à chaque requête)
  ┌─────────────────┬────────────────┬───────────────────────────────────────┐
  │ id              │ UUID (PK)      │                                       │
  │ exercice_id     │ UUID (FK)      │ → exercices_fiscaux.id                │
  │ type_rapport    │ VARCHAR(50)    │ 'bilan', 'resultat', 'sig', 'kpi'     │
  │ donnees         │ JSONB          │ Résultat sérialisé                    │
  │ calcule_le      │ TIMESTAMP      │ Invalidé à chaque nouvel import       │
  └─────────────────┴────────────────┴───────────────────────────────────────┘

  NOTE : Les tables krokmou_conversations, krokmou_messages, krokmou_documents
         existent en base mais ne sont plus utilisées (fonctionnalité supprimée).
         Elles peuvent être supprimées via une migration future.

  noyau                   ← aucune dépendance interne
    ↓
  moteur                  ← dépend de : noyau
    ↓
  api                     ← dépend de : noyau, moteur
    ↓
  interface               ← dépend de : noyau uniquement
                             (ne jamais importer depuis api ou moteur)

  RÈGLE ABSOLUE :
  L'interface ne connait pas le moteur. Elle ne fait que des appels HTTP
  vers l'api. Toute la logique de calcul reste côté serveur.
